## 3. 模块详细设计

### 3.1 配置模块 (`src/config.ts`)

负责从环境变量中读取所有配置项，并提供类型安全的默认值。

```typescript
// 配置接口
interface DeepSeekConfig {
  apiKey: string;           // 必填，来自 DEEPSEEK_API_KEY
  baseUrl: string;          // 默认 "https://api.deepseek.com"
  timeout: number;          // 默认 30000ms
  maxRetries: number;       // 默认 3
}
```

**设计要点：**
- 启动时校验 `DEEPSEEK_API_KEY` 是否存在，缺失则抛出明确错误并退出
- 所有配置项只读，初始化后不可变
- 使用工厂函数 `loadConfig()` 加载，而非全局单例

### 3.2 API 客户端模块 (`src/client.ts`)

封装所有与 DeepSeek API 的 HTTP 通信。

```typescript
class DeepSeekClient {
  private config: DeepSeekConfig;

  constructor(config: DeepSeekConfig);

  // Chat Completion - 普通模式
  async chatCompletion(params: ChatCompletionParams): Promise<ChatCompletionResponse>;

  // Chat Completion - 流式模式
  async chatCompletionStream(params: ChatCompletionParams): Promise<string>;

  // FIM 代码补全
  async fimCompletion(params: FimCompletionParams): Promise<FimCompletionResponse>;

  // 查询模型列表
  async listModels(): Promise<ModelListResponse>;

  // 内部 HTTP 请求方法（带重试和错误处理）
  private async request<T>(endpoint: string, body: unknown): Promise<T>;
  private async requestStream(endpoint: string, body: unknown): Promise<string>;
}
```

**设计要点：**

#### 3.2.1 HTTP 请求实现

- 使用 Node.js 内置 `fetch` API（Node 18+），无需额外依赖
- 请求头固定包含：
  ```
  Authorization: Bearer <api_key>
  Content-Type: application/json
  ```

#### 3.2.2 重试策略

```
重试条件：
  - HTTP 429 (Too Many Requests)
  - HTTP 500 (Internal Server Error)
  - HTTP 502/503/504 (网关错误)
  - 网络连接错误 (ECONNRESET, ETIMEDOUT)

重试策略：指数退避
  第 1 次重试：等待 1000ms
  第 2 次重试：等待 2000ms
  第 3 次重试：等待 4000ms
  最大重试次数：由 config.maxRetries 控制（默认 3）
```

#### 3.2.3 流式处理

```
流式处理流程：
  1. 发送请求，设置 stream: true
  2. 读取 response.body 的 ReadableStream
  3. 使用 TextDecoder 解析每个 SSE 事件
  4. 提取 data: 行中的 JSON 数据
  5. 累积 delta.content 到完整响应
  6. 遇到 data: [DONE] 时结束
  7. 返回累积的完整文本

对于 Reasoner 模型的流式处理：
  - 额外累积 delta.reasoning_content 字段
  - 最终拼接为 "推理过程" + "最终答案" 两部分返回
```

### 3.3 工具处理层

每个工具模块遵循统一的注册模式：

```typescript
// 工具注册模式
function registerXxxTool(server: McpServer, client: DeepSeekClient): void {
  server.tool(
    "tool_name",        // 工具名称
    "工具描述",          // 工具描述
    { /* zod schema */ }, // 参数定义
    async (params) => {   // 处理函数
      // 1. 参数预处理
      // 2. 调用 client 方法
      // 3. 格式化返回结果
      return {
        content: [{ type: "text", text: "..." }],
      };
    }
  );
}
```

#### 3.3.1 Chat 工具 (`src/tools/chat.ts`)

**工具名称：** `deepseek_chat`

**处理流程：**

```
1. 接收参数：message, system_prompt?, model?, temperature?, max_tokens?, top_p?, stream?
2. 构造 messages 数组：
   - 如果有 system_prompt → [{ role: "system", content: system_prompt }]
   - 追加 [{ role: "user", content: message }]
3. 根据 stream 参数：
   - false → 调用 client.chatCompletion()
   - true  → 调用 client.chatCompletionStream()
4. 格式化返回：
   - 非流式：返回 content + token 使用统计
   - 流式：返回累积的完整文本
5. 异常处理：捕获错误，返回 isError: true 的响应
```

**API 请求体映射：**

```json
{
  "model": "deepseek-chat",
  "messages": [
    { "role": "system", "content": "<system_prompt>" },
    { "role": "user", "content": "<message>" }
  ],
  "temperature": 1.0,
  "max_tokens": 4096,
  "top_p": 1.0,
  "stream": false
}
```

**API 端点：** `POST /v1/chat/completions`

#### 3.3.2 Reasoner 工具 (`src/tools/reasoner.ts`)

**工具名称：** `deepseek_reasoner`

**处理流程：**

```
1. 接收参数：message, system_prompt?, max_tokens?, stream?, show_reasoning?
2. 构造请求（model 固定为 "deepseek-reasoner"）
3. 调用 API 获取响应
4. 解析响应中的 reasoning_content 和 content
5. 根据 show_reasoning 参数决定返回格式：
   - true  → 返回 "## 推理过程\n{reasoning}\n\n## 最终答案\n{content}"
   - false → 仅返回 content
6. 附加 token 使用统计
```

**特殊说明：**
- Reasoner 模型不支持 `temperature` 和 `top_p` 参数
- 响应中的 `message.reasoning_content` 为推理链
- 响应中的 `message.content` 为最终答案

#### 3.3.3 FIM 工具 (`src/tools/fim.ts`)

**工具名称：** `deepseek_fim`

**处理流程：**

```
1. 接收参数：prefix, suffix?, max_tokens?, temperature?
2. 构造 FIM 请求体：
   {
     "model": "deepseek-chat",
     "prompt": prefix,
     "suffix": suffix,
     "max_tokens": max_tokens,
     "temperature": temperature
   }
3. 调用 client.fimCompletion()
4. 返回补全的代码文本
```

**API 端点：** `POST /beta/completions`（注意：非 `/v1/` 前缀）

#### 3.3.4 多轮对话工具 (`src/tools/multi-turn.ts`)

**工具名称：** `deepseek_multi_turn`

**处理流程：**

```
1. 接收参数：messages (JSON 数组), model?, temperature?, max_tokens?
2. 验证 messages 格式（每条消息必须有 role 和 content）
3. 直接将 messages 传递给 API
4. 返回助手回复 + token 使用统计
```

**参数验证规则：**
- `messages` 必须是非空数组
- 每个 message 的 `role` 必须是 `system` | `user` | `assistant` 之一
- 每个 message 的 `content` 必须是非空字符串

#### 3.3.5 模型列表工具 (`src/tools/models.ts`)

**工具名称：** `deepseek_list_models`

**处理流程：**

```
1. 无输入参数
2. 调用 client.listModels()
3. 格式化为表格文本返回
```

**API 端点：** `GET /v1/models`

## 4. 类型定义 (`src/types.ts`)

### 4.1 API 请求类型

```typescript
// Chat Completion 请求参数
interface ChatCompletionParams {
  model: string;
  messages: ChatMessage[];
  temperature?: number;
  max_tokens?: number;
  top_p?: number;
  stream?: boolean;
  frequency_penalty?: number;
  presence_penalty?: number;
  stop?: string | string[];
}

// 单条消息
interface ChatMessage {
  role: "system" | "user" | "assistant";
  content: string;
}

// FIM 请求参数
interface FimCompletionParams {
  model: string;
  prompt: string;          // 前缀代码
  suffix?: string;         // 后缀代码
  max_tokens?: number;
  temperature?: number;
  stop?: string | string[];
}
```

### 4.2 API 响应类型

```typescript
// Chat Completion 响应
interface ChatCompletionResponse {
  id: string;
  object: "chat.completion";
  created: number;
  model: string;
  choices: Array<{
    index: number;
    message: {
      role: "assistant";
      content: string;
      reasoning_content?: string;  // 仅 Reasoner 模型
    };
    finish_reason: "stop" | "length" | "content_filter";
  }>;
  usage: TokenUsage;
}

// FIM 响应
interface FimCompletionResponse {
  id: string;
  object: "text_completion";
  choices: Array<{
    index: number;
    text: string;
    finish_reason: "stop" | "length";
  }>;
  usage: TokenUsage;
}

// Token 使用统计
interface TokenUsage {
  prompt_tokens: number;
  completion_tokens: number;
  total_tokens: number;
  prompt_cache_hit_tokens?: number;
  prompt_cache_miss_tokens?: number;
}

// 流式响应 chunk
interface StreamChunk {
  id: string;
  object: "chat.completion.chunk";
  choices: Array<{
    index: number;
    delta: {
      role?: string;
      content?: string;
      reasoning_content?: string;
    };
    finish_reason: string | null;
  }>;
  usage?: TokenUsage;
}

// 模型列表响应
interface ModelListResponse {
  object: "list";
  data: Array<{
    id: string;
    object: "model";
    owned_by: string;
  }>;
}
```

## 5. 错误处理设计 (`src/errors.ts`)

### 5.1 错误分类

```typescript
// 自定义错误基类
class DeepSeekError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly statusCode?: number,
  ) {
    super(message);
    this.name = "DeepSeekError";
  }
}

// 具体错误类型
class AuthenticationError extends DeepSeekError {}    // 401 - API Key 无效
class RateLimitError extends DeepSeekError {}          // 429 - 请求限流
class InsufficientBalanceError extends DeepSeekError {} // 402 - 余额不足
class InvalidRequestError extends DeepSeekError {}     // 400 - 请求参数错误
class ServerError extends DeepSeekError {}             // 500 - 服务端错误
class TimeoutError extends DeepSeekError {}            // 请求超时
class NetworkError extends DeepSeekError {}            // 网络连接失败
```

### 5.2 错误映射表

| HTTP 状态码 | 错误类型 | MCP 返回 | 用户提示 |
|------------|---------|----------|---------|
| 400 | InvalidRequestError | isError: true | "请求参数错误: {detail}" |
| 401 | AuthenticationError | isError: true | "API Key 无效，请检查 DEEPSEEK_API_KEY" |
| 402 | InsufficientBalanceError | isError: true | "账户余额不足，请充值" |
| 429 | RateLimitError | 自动重试 → isError | "请求频率超限，已重试 N 次仍失败" |
| 500 | ServerError | 自动重试 → isError | "DeepSeek 服务器错误，请稍后重试" |
| 502/503/504 | ServerError | 自动重试 → isError | "DeepSeek 服务暂时不可用" |
| 超时 | TimeoutError | isError: true | "请求超时（{timeout}ms），可通过 DEEPSEEK_TIMEOUT 调整" |
| 网络错误 | NetworkError | isError: true | "网络连接失败，请检查网络" |

### 5.3 统一错误处理函数

```typescript
function formatErrorResponse(error: unknown): ToolResponse {
  if (error instanceof DeepSeekError) {
    return {
      content: [{ type: "text", text: `[DeepSeek Error] ${error.message}` }],
      isError: true,
    };
  }
  // 未知错误
  return {
    content: [{ type: "text", text: `[未知错误] ${String(error)}` }],
    isError: true,
  };
}
```

## 6. 流式响应处理设计

### 6.1 SSE 解析流程

```
输入：HTTP Response Body (ReadableStream)
输出：累积的完整文本

处理步骤：
┌────────────────────────────────────────────────┐
│  1. 创建 TextDecoder 和文本缓冲区              │
│  2. 读取 stream chunk                          │
│  3. 将 chunk 解码追加到缓冲区                   │
│  4. 按 "\n\n" 分割缓冲区为事件                  │
│  5. 对每个事件：                                │
│     ├─ 跳过空行和非 "data: " 开头的行           │
│     ├─ 如果是 "data: [DONE]" → 结束循环         │
│     ├─ 解析 JSON → 提取 delta.content           │
│     └─ 累积到结果字符串                         │
│  6. 重复 2-5 直到 stream 结束                   │
│  7. 返回累积的完整文本                          │
└────────────────────────────────────────────────┘
```

### 6.2 Reasoner 流式处理

Reasoner 模型的流式输出分为两个阶段：

```
阶段 1 - 推理阶段：
  delta 中包含 reasoning_content 字段
  content 字段为空

阶段 2 - 回答阶段：
  delta 中 reasoning_content 为空
  content 字段包含最终答案

最终拼接：
  reasoning_buffer = 所有 reasoning_content 累积
  content_buffer = 所有 content 累积
  result = "## 推理过程\n" + reasoning_buffer + "\n\n## 最终答案\n" + content_buffer
```

## 8. 安全设计

### 8.1 API Key 保护

```
1. API Key 仅通过环境变量 DEEPSEEK_API_KEY 传入
2. 日志中对 API Key 进行脱敏处理（仅显示前 8 位 + ***）
3. 错误响应中不包含 API Key 信息
4. Authorization 头仅在 HTTP 请求中使用，不存储到日志
```

### 8.2 日志安全

```
所有日志写入 stderr（不影响 MCP stdio 协议）：
  console.error("[DeepSeek MCP]", message);

日志级别：
  - ERROR: API 调用失败、配置错误
  - WARN:  重试事件、接近限流
  - INFO:  工具调用记录（不含请求/响应体）
  - DEBUG: 完整请求/响应（API Key 脱敏）
```

### 8.3 输入验证

```
所有工具参数通过 zod schema 严格校验：
  - 字符串参数：限制最大长度
  - 数值参数：限制范围（如 temperature: 0-2）
  - 枚举参数：严格匹配允许值
  - messages 数组：校验每个元素的 role 和 content
```

const transport = new StdioServerTransport();
