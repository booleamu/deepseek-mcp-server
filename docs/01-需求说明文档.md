## 1. 项目概述

### 1.1 项目背景

Model Context Protocol (MCP) 是由 Anthropic 推出的开放协议，允许 AI 助手（如 Claude Code）通过标准化接口调用外部工具和服务。DeepSeek 是国内领先的 AI 大模型提供商，提供高性价比、兼容 OpenAI 格式的 API 服务。

本项目旨在开发一个 MCP Server，将 DeepSeek 的全部 API 能力封装为标准 MCP 工具，使用户能够在 Claude Code、Cursor 等支持 MCP 协议的 AI 编辑器中直接调用 DeepSeek 模型。

### 1.2 项目目标

- 封装 DeepSeek API 的全部核心功能为 MCP 工具
- 提供开箱即用的安装和配置体验
- 支持 stdio 和 HTTP 两种 transport 模式
- 兼容 Claude Code、Cursor、Windsurf 等主流 MCP 客户端

### 1.3 目标用户

- 使用 Claude Code 等 MCP 客户端的开发者
- 需要在 AI 工作流中集成 DeepSeek 模型的团队
- 希望利用 DeepSeek 高性价比 API 进行代码生成/推理的用户

## 2. 功能需求

### 2.1 核心功能列表

| 编号 | 功能名称 | 优先级 | 说明 |
|------|---------|--------|------|
| F01 | Chat Completion（对话补全） | P0 | 调用 deepseek-chat 模型进行对话 |
| F02 | DeepSeek Reasoner（深度推理） | P0 | 调用 deepseek-reasoner 模型，返回推理过程和最终答案 |
| F03 | FIM 代码补全 | P1 | Fill-in-the-Middle 代码补全能力 |
| F04 | 流式响应 | P1 | 支持 SSE 流式输出，实时返回生成内容 |
| F05 | 多轮对话管理 | P1 | 支持上下文历史管理的多轮对话 |
| F06 | 模型列表查询 | P2 | 查询当前可用的 DeepSeek 模型列表 |

### 2.2 功能详细描述

#### F01 - Chat Completion（对话补全）

**描述：** 将用户输入的消息发送到 DeepSeek Chat API，获取模型回复。

**输入参数：**

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| message | string | 是 | - | 用户输入的消息内容 |
| system_prompt | string | 否 | "" | 系统提示词，设定模型行为 |
| model | string | 否 | "deepseek-chat" | 模型名称 |
| temperature | number | 否 | 1.0 | 采样温度，范围 0-2 |
| max_tokens | number | 否 | 4096 | 最大输出 token 数 |
| top_p | number | 否 | 1.0 | Top-P 采样参数 |
| stream | boolean | 否 | false | 是否启用流式输出 |

**输出：**
- 模型生成的文本回复
- token 使用统计（prompt_tokens, completion_tokens, total_tokens）

**示例调用：**
```
工具名：deepseek_chat
参数：{ "message": "用 TypeScript 实现一个二叉树", "temperature": 0.7 }
```

#### F02 - DeepSeek Reasoner（深度推理）

**描述：** 调用 DeepSeek-Reasoner (R1) 模型，获取包含完整推理链的回复。

**输入参数：**

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| message | string | 是 | - | 需要推理的问题或任务 |
| system_prompt | string | 否 | "" | 系统提示词 |
| max_tokens | number | 否 | 8192 | 最大输出 token 数 |
| stream | boolean | 否 | false | 是否启用流式输出 |
| show_reasoning | boolean | 否 | true | 是否返回推理过程 |

**输出：**
- `reasoning_content`：模型的逐步推理过程
- `content`：最终答案
- token 使用统计

**示例调用：**
```
工具名：deepseek_reasoner
参数：{ "message": "证明根号2是无理数", "show_reasoning": true }
```

#### F03 - FIM 代码补全

**描述：** Fill-in-the-Middle 模式，根据代码前缀和后缀生成中间代码。

**输入参数：**

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| prefix | string | 是 | - | 代码前缀（光标位置之前的代码） |
| suffix | string | 否 | "" | 代码后缀（光标位置之后的代码） |
| max_tokens | number | 否 | 256 | 最大补全 token 数 |
| temperature | number | 否 | 0 | 采样温度（代码补全通常用低温度） |

**输出：**
- 生成的中间代码片段

**示例调用：**
```
工具名：deepseek_fim
参数：{
  "prefix": "function fibonacci(n: number): number {\n  ",
  "suffix": "\n}\n",
  "max_tokens": 128
}
```

#### F04 - 流式响应

**描述：** 在 F01 和 F02 中启用 `stream: true` 时，以流式方式逐步返回生成内容。

**技术要求：**
- 基于 Server-Sent Events (SSE) 协议
- 实时返回 delta 内容
- 正确处理 `[DONE]` 终止信号
- 对于 Reasoner 模型，同时流式返回 reasoning_content 和 content

#### F05 - 多轮对话管理

**描述：** 支持携带历史消息进行多轮对话，由客户端管理上下文。

**输入参数：**

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| messages | array | 是 | - | 完整的对话历史消息数组 |
| model | string | 否 | "deepseek-chat" | 模型名称 |
| temperature | number | 否 | 1.0 | 采样温度 |
| max_tokens | number | 否 | 4096 | 最大输出 token 数 |

**messages 格式：**
```json
[
  { "role": "system", "content": "你是一个专业助手" },
  { "role": "user", "content": "什么是机器学习？" },
  { "role": "assistant", "content": "机器学习是..." },
  { "role": "user", "content": "能举个具体例子吗？" }
]
```

#### F06 - 模型列表查询

**描述：** 查询当前 DeepSeek 平台可用的模型列表。

**输入参数：** 无

**输出：**
- 可用模型名称列表
- 各模型的上下文长度信息

## 3. 非功能性需求

### 3.1 性能要求

| 指标 | 要求 |
|------|------|
| 工具调用响应时间 | MCP 协议层 < 100ms（不含 API 请求时间） |
| 并发支持 | 支持同一 server 实例处理多个并行工具调用 |
| 内存占用 | 空闲状态 < 50MB |

### 3.2 可靠性要求

- API 调用失败时返回清晰的错误信息（含 HTTP 状态码和错误描述）
- 网络超时默认 30 秒，可配置
- 自动重试机制：429（限流）状态码自动重试，最多 3 次，指数退避

### 3.3 安全性要求

- API Key 通过环境变量传入，不得硬编码或写入日志
- 所有日志输出到 stderr，不污染 stdout（MCP 协议通道）
- 不在响应中暴露 API Key 信息

### 3.4 兼容性要求

| 客户端 | 支持状态 |
|--------|---------|
| Claude Code | 完全支持（stdio） |
| Cursor | 完全支持（stdio） |
| Windsurf | 完全支持（stdio） |
| 自定义 MCP 客户端 | 通过 HTTP transport 支持 |

### 3.5 可维护性要求

- 代码使用 TypeScript 严格模式
- 核心模块单元测试覆盖率 > 80%
- 提供完整的 README 文档和使用示例

## 4. 配置需求

### 4.1 环境变量

| 变量名 | 必填 | 说明 |
|--------|------|------|
| DEEPSEEK_API_KEY | 是 | DeepSeek 平台 API Key |
| DEEPSEEK_BASE_URL | 否 | API 基础地址，默认 `https://api.deepseek.com` |
| DEEPSEEK_TIMEOUT | 否 | 请求超时时间（毫秒），默认 30000 |
| DEEPSEEK_MAX_RETRIES | 否 | 最大重试次数，默认 3 |

### 4.2 Claude Code 集成配置示例

**方式一：项目级 `.mcp.json`**
```json
{
  "mcpServers": {
    "deepseek": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "deepseek-mcp-server"],
      "env": {
        "DEEPSEEK_API_KEY": "sk-your-api-key"
      }
    }
  }
}
```

**方式二：本地开发**
```json
{
  "mcpServers": {
    "deepseek": {
      "type": "stdio",
      "command": "node",
      "args": ["D:/PycharmProjects/deepseek-mcp-server/dist/index.js"],
      "env": {
        "DEEPSEEK_API_KEY": "sk-your-api-key"
      }
    }
  }
}
```

## 5. 约束与限制

### 5.1 DeepSeek API 限制

| 限制项 | 值 |
|--------|------|
| 上下文窗口 | 128K tokens |
| deepseek-chat 最大输出 | 8K tokens |
| deepseek-reasoner 最大输出 | 64K tokens |
| FIM 接口地址 | `/beta/completions`（Beta 阶段） |

### 5.2 MCP 协议限制

- stdio 模式下 stdout 仅用于协议通信，日志必须写 stderr
- 工具返回内容必须为 `content` 数组格式
- 单次工具调用为同步模型（流式在内部处理后一次性返回）
